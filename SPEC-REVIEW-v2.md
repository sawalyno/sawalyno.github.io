## 総評

v1 レビューの 25 件すべてが反映されている。特に以下の改善は質が高い：

- **R-01（CJK 等幅問題）** をターミナル英語限定として §5.3 / §7.1.1 / §13 の 3 箇所で重ね止めしている。仕様としての一貫性が強い。
- **R-02（JSON-LD）** は `legalName` `alternateName` `founder` まで埋めた完全形。法人実在性証明という主目的に対しベストプラクティス。
- **R-04（特商法表記混同回避）** が「サイトに書かない」という消極策で終わらず、§2.2 を「HTML 出力分」と「SPEC 内部メモ」に明確分割し、冒頭の `> 本ドキュメントは社内向けの仕様書` 注記でドキュメント全体の運用ルールに昇格している。リーダブルで運用に効く設計。
- **R-06（クロム要素分離）** が「上＝時計、下＝Start」と一行で記憶できるルールに固まっている。
- **R-25（公開前チェックリスト）** が 7 項目で具体（safe-area・SR で `_ □ ×` が読まれない、まで含む）。
- **付録 A の対応マトリクス**は今後のレビューでも活用しやすい。

その上で v2 仕様で**新たに気付いた論点**と**v1 反映時に副次的に生じた小ズレ**を指摘する。重要度「高」は 1 件、残りは中〜低。スケジュール（5/6 着手）は維持できる程度の編集量。

## 観点別サマリ

### v1 → v2 で確実に良くなったところ
- ドキュメント運用ルール（冒頭 1〜7 行目の `> 本ドキュメントは…`）の追加。**HTML 出力する正は §7 と §9 のみ**、と明示することで、SPEC が長くなっても「どれを実装するか」が誤解されにくい。
- §2.3 のストア表示名 ↔ サイト主体名対応表は、本来口頭の暗黙合意になりがちな部分を 3 行の表に落としていて秀逸。
- §10.2 の Web Vitals 化（LCP / TBT / CLS）は、後の Lighthouse 計測結果を「90+」よりも具体的に評価できる。
- §11.2 手順の AAAA / CAA 追記、§11.3 公開前チェックリストはどちらも「公開後にトラブルが起きやすい場所」を先回り。

### 新たに浮上した論点（v2 ならでは）
- **V2-01**：v1 で時間を分けたことで生じた**算術曖昧性**。仕様書では数字の足し算を読み手にさせない。
- **V2-04**：v1 で導入した HTML エンティティ難読化と、A11y 配慮（読み上げ）のトレードオフ。
- **V2-05**：JSON-LD にコメントが残ったまま。**JSON Lint で fail する**ので、テスト工程で発覚するが、SPEC を見て実装した時点で混入する。
- **V2-07**：CAA 注記は v1 レビューで追加要望したが、**確認手順そのもの**が抜けていた（v1 レビュー側の不足を v2 で補完）。
